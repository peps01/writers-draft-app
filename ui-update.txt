File: src/layouts/MainLayout.vue (UNUSED — legacy boilerplate)

MainAppLayout is the ONLY active layout. It wraps ALL auth-required pages
(and is NOT used on Login/Register/404 pages).

Template structure:
  <q-layout view="hHh LpR lFf" class="wda-dashboard-page">
    <AppNavbar v-if="!layoutFullscreen" />       ← shown on all pages except fullscreen write mode
    <AppSidebar v-if="!layoutFullscreen" />       ← shown on all pages except fullscreen write mode
    <q-page-container>
      <router-view />                             ← page content
    </q-page-container>
    <ExportDialog v-if="currentProjectId" />      ← globally available
  </q-layout>

Key details:
- layoutFullscreen is provided via provide/inject — WritePage toggles it
- currentProjectId is computed from route.params.id
- ExportDialog is rendered at layout level, triggered via inject('openExportDialog')
- Login, Register, and 404 pages do NOT use MainAppLayout — they are standalone

The q-layout has:
  q-header: AppNavbar (64px height)
  q-drawer (left, 220px width, always visible, non-toggleable): AppSidebar
  q-page-container: fills remaining space

3. ROUTER / ROUTES
================================================================================

File: src/router/routes.js
File: src/router/index.js

Mode: hash mode (URLs have /#/)

Route table:
  /                    → redirects to /dashboard
  /login               → LoginPage.vue        [meta: guest only]
  /register            → RegisterPage.vue     [meta: guest only]
  /dashboard           → DashboardPage.vue    [meta: requiresAuth]
  /settings            → SettingsPage.vue     [meta: requiresAuth]
  /projects/:id        → ProjectDetailPage.vue [meta: requiresAuth]
  /projects/:id/story-bible  → StoryBiblePage.vue     [meta: requiresAuth]
  /projects/:id/write        → WritePage.vue          [meta: requiresAuth]
  /projects/:id/statistics   → StatisticsPage.vue     [meta: requiresAuth]
  /:catchAll(.*)*      → ErrorNotFound.vue    [no guard]

Auth-required pages use MainAppLayout (imported in route component-scope).
Guest-only pages and 404 page are standalone (no layout wrapper).

Route guards (in router/index.js beforeEach):
  1. Fetch CSRF token if csrftoken cookie is missing (GET /api/auth/user/ → sets cookie)
  2. If route.meta.requiresAuth && !isAuthenticated → redirect to /login
  3. If route.meta.guest && isAuthenticated → redirect to /dashboard

9. ALL PINIA STORES — COMPLETE STATE & ACTIONS
================================================================================

Each store uses Composition API style (defineStore with setup function).

--- 9.1 AUTH STORE (auth.js) ---
State: user (object|null), loading (boolean)
Getters: isAuthenticated, hasGeminiKey, geminiApiKeyPreview, isPaidTier,
         dailyWordGoal, showWordGoal
Actions:
  login(username, password)          → POST /api/auth/login/
  register(username, email, password) → POST /api/auth/register/
  logout()                           → POST /api/auth/logout/
  fetchCurrentUser()                 → GET /api/auth/user/
  updateProfile(payload)             → PATCH /api/auth/profile/
  testKey()                          → POST /api/auth/profile/test-key/

--- 9.2 PROJECTS STORE (projects.js) ---
State: projects[], loading, error
Actions:
  fetchProjects()                     → GET /api/projects/
  fetchProject(id)                    → GET /api/projects/:id/
  createProject(title)                → POST /api/projects/
  renameProject(id, title)            → PATCH /api/projects/:id/
  deleteProject(id)                   → DELETE /api/projects/:id/

--- 9.3 SCENES STORE (scenes.js) ---
State: scenes[], loading, error, activeSceneId, saveStatus
Actions:
  fetchScenes(projectId)              → GET /api/projects/:pid/scenes/
  createScene(projectId)              → POST /api/projects/:pid/scenes/
  deleteScene(projectId, id)          → DELETE /api/projects/:pid/scenes/:id/
  reorderScenes(projectId, order)     → PATCH bulk reorder
  updateSceneContent(projectId, id, content) → PATCH scene content
  updateSceneMeta(projectId, id, data) → PATCH scene metadata
  setActiveScene(id)

--- 9.4 SCENE VERSIONS STORE (sceneVersions.js) ---
State: versions[], loading, hasMore, restoring, restoreError
Actions:
  fetchVersions(sceneId)              → GET .../versions/
  loadMoreVersions()                  → paginated load
  restoreVersion(verId, password)     → POST .../versions/:vid/restore/
  clearRestoreError()

--- 9.5 CHARACTERS STORE (characters.js) ---
State: characters[], loading, error
Actions:
  fetchCharacters(projectId)          → GET /api/projects/:pid/characters/
  createCharacter(projectId, data)    → POST /api/projects/:pid/characters/
  updateCharacter(projectId, id, data)→ PATCH /api/projects/:pid/characters/:cid/
  deleteCharacter(projectId, id)      → DELETE /api/projects/:pid/characters/:cid/

--- 9.6 PLACES STORE (places.js) ---

12. DATA FLOW & STATE MANAGEMENT PATTERNS
================================================================================

Authentication flow:
  1. App loads → beforeEach route guard checks for csrftoken cookie
  2. If missing, fetches /api/auth/user/ (which sets the cookie via response)
  3. If authenticated, authStore.user is populated; if not, redirect to /login
  4. All mutating requests include X-CSRFToken header from cookie (axios interceptor)
  5. 401/403 responses → auto-redirect to /login (axios response interceptor)

Editor auto-save:
  1. User stops typing for 2.5s
  2. saveSceneContent() called → updates scene content via PATCH
  3. saveStatus state updated: "Saving..." → "Saved" → "Ready"
  4. Auto-tagging runs every 3s: scans ProseMirror content for entity names

Data fetching pattern:
  - Pages fetch data on mount via stores
  - Loading/error states displayed while fetching
  - Entities cached in Pinia stores (no refetch unless explicitly called)

Dialog patterns:
  - Simple dialogs (confirm delete, etc.): inline in page components
  - Complex dialogs (EntityFormDialog, ExportDialog): separate components
  - Global dialog (ExportDialog): rendered in MainAppLayout, triggered via inject

  - Floating action button pattern for primary actions

    - Or vertical list with compact cards
    - Skeleton states need mobile sizing

    - Editor needs full-width with minimal chrome
    - Formatting toolbar: collapsible or bottom toolbar
    - Find & Replace: bottom sheet

    - Story Bible tags: collapsible accordion at bottom
    - Fullscreen mode: hide ALL chrome, swipe to switch scenes
